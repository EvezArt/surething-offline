# Operator API Contract

Status: current observed contract for the live EVEZ operator spine.

Repository role: `surething-offline` acting as the logical **`evez-operator-api`**.
Deployment label: `evez-operator`.

This document describes the current HTTP contract exposed by `api/index.py`.
It is intentionally explicit about what is authoritative, what is temporary, and what should not yet be treated as durable truth.

---

## Contract posture

### Authoritative today
These are the parts of the API that are structurally real and should be treated as the current source of truth for clients:

- route existence and HTTP verbs
- request/response shapes defined in `api/index.py`
- event canonicalization behavior
- event hash / evidence hash generation behavior
- lane and event-type validation
- spine parent-chain validation for the current in-memory process

### Not yet durable / not yet authoritative across restarts
These behaviors are real in-process but should **not** be treated as durable system-of-record behavior yet:

- `_spine` storage is in-memory only
- spine history is lost on cold start / redeploy / process restart
- contradiction rows are currently static return values
- deploy-status rows are currently static or semi-static return values
- no authentication or write protection is present yet
- no persistence backend is wired for operator events yet

### Client rule
Clients may safely integrate against the route shapes below, but must treat several payloads as `ephemeral`, `static`, or `diagnostic` until persistence/auth/state backends are added.

---

## Base assumptions

- Content type: `application/json` unless otherwise stated
- Timestamps: RFC3339 UTC with `Z` suffix where generated by event creation
- Canonicalization: NFC string normalization + sorted keys + strict type handling + SHA-256
- CORS: permissive (`*`) in current implementation

## Route index

| Route | Method | Purpose | Authority level |
|---|---|---|---|
| `/` | GET | basic service identity | authoritative |
| `/tools` | GET | MCP/tool manifest | authoritative |
| `/spine/event` | POST | append typed event to current process spine | authoritative shape, ephemeral storage |
| `/spine/verify` | GET | verify parent hash chain of current process spine | authoritative shape, ephemeral scope |
| `/spine/events` | GET | list current process spine events | authoritative shape, ephemeral scope |
| `/canonicalize` | POST | canonicalize and hash a JSON object | authoritative |
| `/contradictions` | GET | list unsat cores | shape authoritative, payload currently static |
| `/fire/anchor` | POST | generate Base Sepolia calldata payload | authoritative |
| `/deploy/status` | GET | service-health view | shape authoritative, payload currently static/diagnostic |

## 1) `GET /`

### Purpose
Service discovery and shallow health.

### Response
```json
{
  "status": "ok",
  "service": "evez-operator-api",
  "version": "1.0.0",
  "spine_height": 0,
  "docs": "/docs",
  "tools": "/tools"
}
```

### Field notes
- `status`: authoritative string for shallow readiness only
- `service`: authoritative service identifier
- `version`: authoritative API implementation version string
- `spine_height`: authoritative only for the current in-memory process
- `docs`, `tools`: authoritative route hints

## 3) `POST /spine/event`

### Purpose
Append a typed operator event to the current process spine.

### Request body
```json
{
  "event_type": "claim",
  "lane": "empirical",
  "payload": {
    "subject": "queue-health",
    "status": "degraded"
  },
  "parent_event_hash": "optional-previous-hash"
}
```

### Allowed `event_type`
- `observation`
- `transformation`
- `claim`
- `test`
- `decision`
- `linkage`

### Allowed `lane`
- `empirical`
- `documentary`
- `metaphysical`

### Success response
```json
{
  "ok": true,
  "event_hash": "...",
  "evidence_hash": "...",
  "height": 0
}
```

### Error responses
Invalid `event_type` or `lane` returns HTTP 400 with detail text.

Representative examples:
```json
{
  "detail": "Invalid event_type: foo"
}
```

```json
{
  "detail": "Invalid lane: bar"
}
```

### Authority notes
- `event_hash`, `evidence_hash`, and `height` are authoritative for the current process state
- persistence is **not** durable across restart
- if `parent_event_hash` is omitted, the implementation auto-links to the current in-memory tip if present

### Client cautions
Do not assume this is a permanent append-only ledger yet. Today it is an ephemeral process ledger with correct hashing semantics.

## 4) `GET /spine/verify`

### Purpose
Verify the parent-chain integrity of the current process spine.

### Query params
- `from_height` (integer, default `0`)

### Success response when valid
```json
{
  "valid": true,
  "checked": 4,
  "tip": "latest-event-hash-or-null"
}
```

### Broken-chain response
```json
{
  "valid": false,
  "broken_at": 3,
  "expected": "hash-a",
  "got": "hash-b"
}
```

### Authority notes
- verification logic is authoritative
- scope is only the current in-memory event list

## 6) `POST /canonicalize`

### Purpose
Normalize and hash a JSON object according to the current canonicalization rules.

### Request body
```json
{
  "input": {
    "hello": "world"
  },
  "float_paths": [".score"]
}
```

### Success response
```json
{
  "jcs": "{\"hello\":\"world\"}",
  "sha256": "...",
  "bytes": 17
}
```

### Error behavior
Returns HTTP 400 if:
- a float appears on a non-declared path
- a float is NaN or Infinity
- unsupported value types are included

Representative error shape:
```json
{
  "detail": "Float at non-declared path .score"
}
```

### Authority notes
This endpoint is one of the strongest current primitives in the API and can be treated as authoritative.

## 8) `POST /fire/anchor`

### Purpose
Produce a canonicalized payload and calldata blob for FIRE anchoring.

### Current parameter mode
The current implementation expects request values as function parameters in FastAPI, which typically means query/form-style submission unless refactored to a body model.

Required:
- `fire_number` (integer)
- `evidence_hash` (string)

Optional:
- `round` (integer, default `0`)
- `v_score` (float, default `0.0`)

### Success response
```json
{
  "fire_number": 125,
  "anchor_hash": "...",
  "calldata": "0x...",
  "calldata_length_bytes": 123,
  "network": "base-sepolia"
}
```

### Authority notes
- payload generation and hashing are authoritative
- `network` is authoritative as currently coded (`base-sepolia`)
- this endpoint does **not** submit a transaction; it only prepares the payload

### Client caution
Treat this as a build/prepare endpoint, not a chain-write endpoint.

## Common event object

When an event is returned from spine routes, the current shape is:

```json
{
  "event_type": "claim",
  "lane": "empirical",
  "payload": {},
  "timestamp": "2026-03-19T06:00:00Z",
  "height": 0,
  "parent_event_hash": "",
  "evidence_hash": "...",
  "event_hash": "..."
}
```

### Field meanings
- `event_type`: semantic class of event
- `lane`: epistemic/operator lane
- `payload`: caller-supplied JSON payload
- `timestamp`: server-generated event creation time
- `height`: zero-based index within current process spine
- `parent_event_hash`: previous event pointer, empty string for genesis in current implementation
- `evidence_hash`: canonical hash of event without ancestry field
- `event_hash`: canonical hash of full event including ancestry field

## Error contract

The API currently relies on FastAPI default error responses for many failures.

Typical error shape:
```json
{
  "detail": "human-readable message"
}
```

### Expected error classes
- HTTP 400 for invalid lane/event_type/canonicalization input
- HTTP 422 for missing required parameters or invalid FastAPI coercion

Clients should not overfit to exact wording of `detail` strings.

---

## Current limitations that client builders must respect

1. No persistence backend for `_spine`
2. No auth
3. No rate limiting
4. No write protection
5. No externally verified deploy-health probes
6. Contradictions are static rows, not dynamic unsat derivations yet
7. `fire/anchor` prepares payloads only; it does not submit or confirm chain writes

---

## Recommended next contract upgrades

### Upgrade 1 — persistence
Move `_spine` to durable storage and preserve `height`, `tip`, and event chain across restarts.

### Upgrade 2 — explicit request models
Add body models for endpoints like `/fire/anchor` to remove ambiguity around parameter transport.

### Upgrade 3 — health probing
Replace static `deploy/status` rows with active checks and machine-readable probe results.

### Upgrade 4 — contradiction source-of-truth
Move contradiction rows from static-coded values to spine-backed or registry-backed state.

### Upgrade 5 — auth
Add operator authentication before exposing write endpoints to real clients.

---

## Integration guidance for the mobile operator console

The mobile console may safely bind to this API today if it follows these rules:

- treat route shapes as real
- treat event hashing/canonicalization as real
- treat contradiction and deploy rows as diagnostic placeholders unless verified elsewhere
- label current spine state as `ephemeral` or `session-bound`
- never present current spine storage as durable history to users

That lets the console stop faking telemetry while still remaining honest about what the operator spine actually guarantees today.
